### Orphaned Exports

**CLAUDE_WATCHDOG_HOURS** (in `systemd/env.template`, line 24)
- Set to `8` as a documentation variable
- The deployed `claude-watchdog@.timer` hardcodes `OnBootSec=8h` and `OnUnitActiveSec=8h`
- systemd timer directives cannot read environment files; hardcoding is the correct workaround (confirmed in Phase 8-01 key-decisions)
- **Classification:** Intentional orphan. The variable serves as operator-visible documentation of the intended interval. It is not a bug.
- **Affected requirements:** None (WDOG-04 covers template structure, not configurability)

### Missing Connections

**orchestra/CLAUDE.md is not auto-deployed by `claude-service add-orchestra`**
- `do_add_orchestra` creates `~/instruments/orchestra/` directory and provisions env, systemd units, and .mcp.json
- The function ends with: `echo "Next: place orchestra CLAUDE.md at $work_dir/CLAUDE.md"` — a manual instruction, not an automated copy
- The `orchestra/CLAUDE.md` file exists in the repo at `/orchestra/CLAUDE.md` and is not copied to the instrument working directory automatically
- **Impact:** The orchestra service starts correctly as a systemd unit, but on first session launch it has no CLAUDE.md and therefore no behavioral specification. The orchestra session will start as a generic Claude session.
- **Affected requirements:** ORCH-01 (orchestra is an instrument with CLAUDE.md), ORCH-02 (dispatches one-shot agents), ORCH-03 (restarts instruments), ORCH-05 (routes messages)
- **Severity:** Medium. VPS deployment works but requires a manual step that is not scripted. First-time user error risk.

## Wiring Detail: NODEVERSION_PLACEHOLDER

`env.template` contains `NODEVERSION_PLACEHOLDER` in the PATH line. `install.sh` substitutes this (or strips the nvm segment) during fresh install. `claude-service add` and `add-orchestra` do NOT substitute `NODEVERSION_PLACEHOLDER` — they instead overwrite the entire PATH line with the value copied from the default instance env (lines 98-101 and 162-166 of claude-service).

This means: if `grep '^PATH=' "$default_env"` returns empty (which cannot happen after a correct install, since install.sh always writes a PATH line), the new instrument env would contain a literal `NODEVERSION_PLACEHOLDER` in PATH and `~/instruments/<name>` would not be in PATH. This is a theoretical edge case, not a real risk for correctly installed systems. No action required.

## Findings Requiring Action

### FINDING-01: orchestra/CLAUDE.md not auto-deployed (Medium)

**Affected requirements:** ORCH-01, ORCH-02, ORCH-03, ORCH-05
**Break point:** Flow 2, Step 2 — after `claude-service add-orchestra` completes, the orchestra service starts but has no behavioral spec until the user manually copies the file.
**Current behavior:** `do_add_orchestra` prints `Next: place orchestra CLAUDE.md at ~/instruments/orchestra/CLAUDE.md` and exits.
**Expected behavior for zero-friction E2E:** `do_add_orchestra` should copy `orchestra/CLAUDE.md` to `~/instruments/orchestra/CLAUDE.md` during provisioning.
**Risk:** User who follows the `add-orchestra` path without reading the output carefully will have an orchestra session with no behavioral spec. The session will start, consume API credits, and do nothing useful.
**Suggested fix:** Add `cp "$SCRIPT_DIR/../orchestra/CLAUDE.md" "$work_dir/CLAUDE.md"` to `do_add_orchestra`, falling back to the manual instruction if the source file is not found.
