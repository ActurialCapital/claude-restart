<objective>
Fix the blocker where orchestra remote-control mode starts but never spawns a session. The root cause (from debug session .planning/debug/remote-control-no-session.md) is that the "Enable Remote Control?" confirmation prompt may not reliably consume the FIFO-piped "y" due to readline timing/buffering issues with non-TTY stdin. The fix: pre-set `remoteDialogSeen: true` in claude's global config (`~/.claude.json`) before launching, bypassing the prompt entirely. Then remove the now-unnecessary `echo "y"` from the FIFO writer.

**1. Add a helper function to bin/claude-wrapper** (near the top, after variable declarations, before the while loop):

```bash
# Ensure remoteDialogSeen is set in claude global config (bypasses "Enable Remote Control?" prompt)
ensure_remote_dialog_seen() {
    local config_file="$HOME/.claude.json"
    if [[ -f "$config_file" ]]; then
        # Check if remoteDialogSeen is already true
        if command -v jq &>/dev/null; then
            if [[ "$(jq -r '.remoteDialogSeen // false' "$config_file" 2>/dev/null)" == "true" ]]; then
                return 0
            fi
            # Set it using jq
            local tmp
            tmp=$(jq '.remoteDialogSeen = true' "$config_file") && echo "$tmp" > "$config_file"
        elif command -v python3 &>/dev/null; then
            python3 -c "
import json, sys
with open('$config_file') as f: d = json.load(f)
if d.get('remoteDialogSeen') == True: sys.exit(0)
d['remoteDialogSeen'] = True
with open('$config_file', 'w') as f: json.dump(d, f, indent=2)
"
        else
            echo "claude-wrapper: warning: cannot set remoteDialogSeen (install jq or python3)" >&2
        fi
    else
        # Create minimal config
        echo '{"remoteDialogSeen": true}' > "$config_file"
    fi
}
```

**4. In bin/install.sh**, add the remoteDialogSeen pre-set AFTER the env file creation block (after the closing `fi` on line 142) but BEFORE systemd daemon-reload. IMPORTANT: Do NOT use the interactive `CONN_MODE` variable -- it is only set inside the interactive env creation branch and would be empty for non-interactive deploys (when env file already exists). Instead, read the CLAUDE_CONNECT value from the written env file:

```bash
# Pre-set remoteDialogSeen in claude global config for non-interactive remote-control startup
# Read CLAUDE_CONNECT from the env file (works for both fresh installs and re-runs)
EFFECTIVE_CONN_MODE=$(grep '^CLAUDE_CONNECT=' "$ENV_FILE" 2>/dev/null | cut -d= -f2 || echo "")
if [[ "$EFFECTIVE_CONN_MODE" == "remote-control" ]]; then
    CLAUDE_CONFIG="$HOME/.claude.json"
    if [[ -f "$CLAUDE_CONFIG" ]] && command -v jq &>/dev/null; then
        tmp=$(jq '.remoteDialogSeen = true' "$CLAUDE_CONFIG") && echo "$tmp" > "$CLAUDE_CONFIG"
        echo "Set remoteDialogSeen=true in $CLAUDE_CONFIG"
    elif [[ ! -f "$CLAUDE_CONFIG" ]]; then
        echo '{"remoteDialogSeen": true}' > "$CLAUDE_CONFIG"
        echo "Created $CLAUDE_CONFIG with remoteDialogSeen=true"
    else
        echo "Warning: install jq to auto-set remoteDialogSeen (or run 'claude' interactively once)"
    fi
fi
```

This placement (after the env file block's closing `fi`, before systemd daemon-reload) ensures:
- Fresh installs: env file was just created with CLAUDE_CONNECT set, grep reads it
- Re-installs (env file exists, skipped): env file already has CLAUDE_CONNECT, grep reads it
- Non-remote-control modes: grep returns "telegram" or empty, condition skips

Without a timeout, `head -1` would block until the heartbeat fires (default 300s). Also set `CLAUDE_WRAPPER_HEARTBEAT_INTERVAL=1` in the test environment so a heartbeat arrives within the timeout window. The rewritten mock:

```bash

#!/bin/bash
echo "\$@" >> "$LOG"
# Read first line from stdin with timeout -- should NOT be "y" anymore
read -t 2 line < /dev/stdin 2>/dev/null && echo "\$line" > "$STDIN_LOG" || true
# Drain remaining stdin to prevent FIFO blocking
cat > /dev/null &
_drain_pid=\$!
kill \$_drain_pid 2>/dev/null; wait \$_drain_pid 2>/dev/null || true
exit 0
MOCKEOF
```
