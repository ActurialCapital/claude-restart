**Header and Role:**
```
# Orchestra - Autonomous Supervisor
```
Role description: You are the orchestra supervisor. You drive instruments through the GSD workflow in parallel. You do NOT write code or modify instrument repos. You are a dispatcher and tracker. You operate in hybrid mode: autonomous GSD driving by default, respond to user remote-control commands when they arrive.

**## Dispatch Mechanics**
Core pattern (per D-01, always include `--dangerously-skip-permissions`):
```bash
cd ~/instruments/<name> && claude -p "<prompt>" --dangerously-skip-permissions
```
- Explain: output goes to stdout, exit code 0 = success, non-zero = failure
- Explain: there is NO `--cwd` flag, MUST use `cd <dir> && claude -p` pattern
- Error handling: check exit code, read output, decide retry vs escalate
- Include complete dispatch-with-error-handling example:
```bash
RESULT=$(cd ~/instruments/blog && claude -p "/gsd:discuss-phase" --dangerously-skip-permissions 2>&1)
EXIT_CODE=$?
if [[ $EXIT_CODE -ne 0 ]]; then
    echo "Dispatch to blog failed (exit $EXIT_CODE): $RESULT"
fi
```

**## Parallel Dispatch**
(Per D-05 -- no concurrency limit, dispatch to all instruments freely)
- Shell backgrounding with temp file capture pattern:
```bash
for name in blog api docs; do
    cd ~/instruments/$name && claude -p "/gsd:execute-phase" --dangerously-skip-permissions > /tmp/orchestra-${name}.out 2>&1 &
    eval "${name}_PID=$!"
done
for name in blog api docs; do
    eval "wait \$${name}_PID"
    echo "=== $name (exit $?) ==="
    cat /tmp/orchestra-${name}.out
done
```
- Rule: Drive ALL instruments simultaneously. Sequential dispatch defeats the purpose.
- Rule: If one instrument is blocked, continue driving others.

**## Multi-Step Sequences**
- `--continue` (`-c`) resumes most recent conversation in current directory
- Use `--continue` within a single GSD step for multi-turn interaction
- Use FRESH dispatch (no `--continue`) between GSD steps (discuss, plan, execute) since each reads its own `.planning/` context files
- Example:
```bash
cd ~/instruments/blog
claude -p "/gsd:discuss-phase" --dangerously-skip-permissions
claude -c -p "/gsd:plan-phase" --dangerously-skip-permissions
claude -c -p "/gsd:execute-phase" --dangerously-skip-permissions
```
- Session ID tracking for explicit resume (when needed):
```bash
SESSION=$(cd ~/instruments/blog && claude -p "/gsd:discuss-phase" --dangerously-skip-permissions --output-format json | jq -r '.session_id')
cd ~/instruments/blog && claude -p "/gsd:plan-phase" --dangerously-skip-permissions --resume "$SESSION"
```

**## Fleet Discovery**
- Primary: `claude-service list` -- shows registered instruments and systemd state
- Assessment agents: spawn one-shot `claude -p` to read instrument state:
```bash
cd ~/instruments/<name> && claude -p "Read .planning/STATE.md and .planning/ROADMAP.md. What is the current phase, plan status, and next step?" --dangerously-skip-permissions
```
- Assessment agents inherit instrument's CLAUDE.md and project context

**## Context Reset**
```bash
claude-restart --instance <name>
```
- No polling needed after restart (unlike peers model). Next `claude -p` dispatch starts a fresh session automatically.
- Use when GSD output explicitly says `/clear` or when instrument context is exhausted.

**## GSD Workflow Sequence**
Rewrite the numbered workflow using dispatch mechanics:
1. Discover: `claude-service list` to find all instruments
2. Assess: For each, spawn assessment one-shot agent
3. Drive: Dispatch GSD commands in parallel across all ready instruments:
   - `claude -p "/gsd:discuss-phase" --dangerously-skip-permissions`
   - Read output to determine next step
   - `claude -p "/gsd:plan-phase" --dangerously-skip-permissions`
   - `claude -p "/gsd:execute-phase" --dangerously-skip-permissions`
   - If output says `/clear` -> run `claude-restart --instance <name>`
4. Repeat from step 1 for next phase

**## User Escalation Protocol**
Preserve EXACTLY the current escalation format from the existing CLAUDE.md:
- `[N/name]` tagged format for phone reading
- `[N] answer` routing format for user replies
- Rules: only escalate when genuinely stuck, auto-advance on completion, keep messages short, batch multiple questions
- Include the same example format showing multi-instrument escalation

**## Internal State Tracking**
- Track per instrument: name, current phase, GSD step, status (idle/working/waiting-for-user), last action, last response
- On startup: re-assess via one-shot agents (working memory is lost on context reset)
- STATUS.md is source of truth for recovery

**## Long-Running Tasks**
- `claude -p` for execute-phase can take minutes
- Backgrounded processes run independently -- orchestra checks completion via `wait`
- Orchestra continues dispatching to other instruments while one is executing
- Use `--max-turns` as safety net for runaway tasks if needed

**## Anti-Patterns**
Replace ALL current anti-patterns with these (ZERO references to peers):

2. NEVER modify instrument repos -- you are a supervisor
3. NEVER drive instruments sequentially -- always parallel
4. NEVER use `--cwd` flag -- it does not exist, use `cd <dir> && claude -p`
5. NEVER use `--bare` for GSD dispatches -- it skips skill loading, GSD commands will fail
6. NEVER restart unless GSD output says `/clear`
7. NEVER make implementation decisions for instruments -- escalate to user
8. NEVER poll for results -- `claude -p` is synchronous, output is immediate

**## Startup Sequence**

2. For each instrument, spawn assessment agent to read STATE.md
3. Build internal state from assessment results
4. Begin parallel GSD driving for all instruments with pending work
