Create `systemd/claude@.service` (new file, do NOT modify `claude.service` -- it stays as reference for migration).

The template unit uses `%i` (instance name) and `%h` (home directory) specifiers. Content:

```ini
[Unit]
Description=Claude Code AI Assistant (%i)
After=network-online.target
Wants=network-online.target
StartLimitBurst=5
StartLimitIntervalSec=60

[Service]
Type=simple
ExecStart=%h/.local/bin/claude-wrapper --dangerously-skip-permissions
EnvironmentFile=%h/.config/claude-restart/%i/env
WorkingDirectory=%h
Restart=on-failure
RestartSec=5
KillSignal=SIGTERM
TimeoutStopSec=10
MemoryMax=%E/claude-restart/%i/memory_max

[Install]
WantedBy=default.target
```

IMPORTANT: systemd cannot expand environment variables in `MemoryMax=` (it is a unit directive, not a shell command). Instead, use a workaround: set `MemoryMax=` to a value read from an override. The cleanest approach per D-09/D-10 is to NOT put MemoryMax directly in the template. Instead, the install script will create a systemd drop-in override per instance, OR we can use `systemd-run` approach.

Actually, the simplest approach that avoids drop-ins (per D-10 "no systemd drop-ins needed") is: Use `ExecStartPre` to validate, and set a fixed default in the template with documentation that per-instance overrides use `systemctl --user edit claude@<name>`. BUT the user said no drop-ins.

Revised approach: Use `MemoryMax=` with a fixed default of `1G` in the template unit. Per-instance memory customization is a Phase 8+ concern. The env file will document `CLAUDE_MEMORY_MAX` for future use, but the template unit uses a hardcoded `1G` since systemd unit directives cannot read env vars.

Wait -- re-reading D-09 and D-10: "MemoryMax is configured via CLAUDE_MEMORY_MAX env var" and "Template unit reads MemoryMax=${CLAUDE_MEMORY_MAX}". The user's intent is clear. Let me check if systemd supports this.

systemd DOES NOT support `MemoryMax=${CLAUDE_MEMORY_MAX}` -- environment variable substitution only works in `Exec*` directives, not in resource control directives like `MemoryMax`.

The correct implementation: Use `ExecStartPre` to create a per-instance cgroup memory limit via a small helper, OR accept that `MemoryMax=1G` is hardcoded and document it. Since D-10 says "no systemd drop-ins needed", the best approach is:

1. Set `MemoryMax=1G` as default in the template
2. Add a comment: "Override per-instance with: systemctl --user set-property claude@<name>.service MemoryMax=2G"
3. The env var `CLAUDE_MEMORY_MAX` in the env template serves as documentation of the intended limit

Actually, the most elegant solution: Use `ExecStartPre` with `systemctl --user set-property` to read the env var and apply it dynamically:

```ini
ExecStartPre=/bin/bash -c 'systemctl --user set-property claude@%i.service MemoryMax=${CLAUDE_MEMORY_MAX:-1G} 2>/dev/null || true'
```

This reads the env var (which IS available in ExecStartPre since EnvironmentFile is loaded first) and applies it as a runtime property. No drop-ins needed. Falls back to 1G if unset.

Final template unit content:

```ini
[Unit]
Description=Claude Code AI Assistant (%i)
After=network-online.target
Wants=network-online.target
StartLimitBurst=5
StartLimitIntervalSec=60

[Service]
Type=simple
EnvironmentFile=%h/.config/claude-restart/%i/env
ExecStartPre=/bin/bash -c 'systemctl --user set-property claude@%i.service MemoryMax=${CLAUDE_MEMORY_MAX:-1G} 2>/dev/null || true'
ExecStart=%h/.local/bin/claude-wrapper --dangerously-skip-permissions
WorkingDirectory=${WORKING_DIRECTORY}
Restart=on-failure
RestartSec=5
KillSignal=SIGTERM
TimeoutStopSec=10

[Install]
WantedBy=default.target
```

Key differences from claude.service:
- `Description` includes `(%i)` for instance identification
- `EnvironmentFile` uses `%i` instead of flat path: `%h/.config/claude-restart/%i/env`
- `WorkingDirectory` uses `${WORKING_DIRECTORY}` env var (per D-04) instead of hardcoded placeholder
- `ExecStartPre` applies MemoryMax from `CLAUDE_MEMORY_MAX` env var (per D-09, D-10)
- No `WORKING_DIR_PLACEHOLDER` -- uses env var instead
