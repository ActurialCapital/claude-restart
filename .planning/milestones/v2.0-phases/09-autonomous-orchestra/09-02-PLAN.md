**Section 3: GSD Workflow Sequence**

Per D-04, define the exact sequence. For each instrument with pending work:

```
1. Discover: list_peers(scope: "machine") to find all instruments
2. Assess: For each instrument, spawn one-shot agent:
   cd ~/instruments/<name> && claude -p "Read .planning/STATE.md and .planning/ROADMAP.md. What is the current phase, plan status, and next step?" --dangerously-skip-permissions
3. Drive: Send GSD commands via send_message:
   a. send_message(to_id, "/gsd:discuss-phase") -- wait for response
   b. If response needs user input -> escalate (see User Escalation)
   c. send_message(to_id, "/gsd:plan-phase") -- wait for response
   d. send_message(to_id, "/gsd:execute-phase") -- wait for response
   e. If response contains "/clear" -> claude-restart --instance <name>
   f. Poll list_peers until instrument re-registers
   g. Continue with next GSD command
4. Repeat from step 1 for next phase
```

**Section 4: Parallel Dispatch**

Per D-09, orchestra drives ALL instruments simultaneously:
- Send GSD commands to all ready instruments at the same time
- Do NOT wait for instrument A to finish before starting instrument B
- Process responses as they arrive via check_messages
- Track each instrument's state independently

**Section 5: User Escalation Protocol**

Per D-10, numbered/tagged format for phone-readable escalation:
```
I need your input on multiple instruments:

[1/blog] The discuss phase identified two possible directions for the blog redesign:
  a) Full rewrite with new framework
  b) Incremental migration
  Which do you prefer?

[2/api] The API project has a failing test in auth module. Should I:
  a) Have the instrument fix it before continuing
  b) Skip and continue to next phase

Reply with the tag to route your answer, e.g.: [1] b
```

Rules:
- Only escalate when genuinely stuck or ambiguous (per D-16)
- Auto-advance on completion without asking (per D-16)
- Keep escalation messages short (user reads on phone)
- Use `[N/name]` prefix for each instrument question
- Accept `[N] answer` format for routing replies

**Section 6: Internal State Tracking**

Per D-07, orchestra tracks state internally (not via STATE.md):
- Maintain a mental model of each instrument: name, current phase, current GSD step, last action, status (idle/working/waiting-for-user/restarting)
- Update state after each interaction
- When user asks "what's happening" -- summarize all instrument states

**Section 7: Anti-Patterns (Things to NEVER Do)**

Include these exact anti-patterns:
1. NEVER read files in instrument repos directly (use claude -p one-shot agents)
2. NEVER modify instrument repos or configs
3. NEVER drive instruments sequentially -- always parallel
4. NEVER cache peer IDs across restarts -- always re-discover via list_peers
5. NEVER restart an instrument unless GSD output explicitly says /clear
6. NEVER make implementation decisions for instruments -- they hold project intelligence
7. NEVER use --cwd flag (does not exist) -- use cd + claude -p pattern

**Section 8: Startup Sequence**

When orchestra first starts or after its own context reset:
1. `set_summary(summary: "Orchestra starting up")`
2. `list_peers(scope: "machine")` to discover all instruments
3. For each instrument, spawn assessment agent:
   `cd ~/instruments/<name> && claude -p "Read .planning/STATE.md. What is current phase and status?" --dangerously-skip-permissions`
4. Build internal state from assessment results
5. Begin parallel GSD driving for all instruments with pending work
6. `set_summary(summary: "Driving N instruments")` with actual count
